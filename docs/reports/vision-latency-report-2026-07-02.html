AprilTag docking on OpenAMRobot — the real blocker is vision latency, not the control

Engineering note for Alex & Raj · 2026-07-02 · Platform: Raspberry Pi 5 (4 cores), Sony IMX708 @ 1280×720, RPLIDAR A1, Teensy (micro-ROS), ROS 2 Jazzy, Nav2, AprilTag 36h11 bundle as the dock target.

TL;DR. The docking sequencer and the drivetrain are fine. We spent the day chasing control symptoms (the visual servo oscillates, drifts into the wrong tag, loses lock). The real cause is upstream: the AprilTag detector hands the servo tag positions that are 250–400 ms stale (were 700 ms and growing). A servo fed stale data chases its target and becomes unstable. The staleness is CPU saturation: the Pi 5 sits at load ≈ 8 on 4 cores when it runs Nav2 + the camera + full-resolution AprilTag at the same time. Fix the CPU budget before touching the control gains again.

1. What we did today

  • Characterised the drivetrain. Open-loop + closed-loop velocity sweeps on the ground. Result: the motor is healthy and well-sized; the low-speed judder is stick-slip below measured floors, not a fault.
  • Rebuilt the Phase-5 visual corrector — camera-frame PD on the centre-tag bearing, made dominant over the noisy map-frame line, with a hysteresis deadband, coast-through-dropout, and odometry dead-reckoning of the final blind centimetres.
  • Investigated the LiDAR (IR-dot-on-tags hypothesis) — dead end (see §2).
  • Built diagnostics and found the root cause: detection latency from CPU saturation (§3).
  • Deployed the first fixes: AprilTag decimate 2.0, and a frame-rate throttle in the image gate (§5).

2. The problems we hit — with root cause

2.1  resolved  Motor "struggles at low speed"

Suspected a bad wheel or an under-sized motor. Measured instead:

  • Left wheel healthy (10.4 vs 10.9 rpm, no dropouts).
  • Linear floor 0.04 m/s reliable / 0.05 clean; angular floor 0.15 rad/s. Below → stick-slip stall.
  • Motor Z4BLD60-24GN-30S: 60 W, 30:1 gearbox, ~100 rpm wheel (~0.49 m/s max), ~4.18 N·m/wheel — over-sized on torque.

The floors are stick-slip + coarse Hall commutation at low RPM (running a geared BLDC at ~10–30 % of max speed), not a torque shortfall. Docking speeds were re-based above the floors.

2.2  downstream of §3  Visual servo instability

Drove into the outer tag (id 0) instead of the centre (id 1); left-right oscillation; "aligns but advances poorly". We rebuilt the corrector (PD + hysteresis + coast + odom dead-reckoning) — better, but the instability persisted, because good control on 300 ms-old data still chases. This pointed us upstream.

2.3  dead end  The LiDAR IR-dot idea

We wired the sequence to stop the LiDAR at the far→near hand-over (the near phase is camera-only). Result: stopping the LiDAR motor killed the AprilTag detection within ~1 s, reproducibly — most likely the rplidar node and apriltag share a process container. Reverted (behind a default-off parameter). Caveat: because the cut broke detection for an unrelated reason, we never actually got to test the IR-dot hypothesis — whether the IR dot also degrades detection remains open. It may well be a compound problem; what we have measured and know to be dominant is the latency, so the IR dot, if it contributes, is secondary.

2.4  open  Camera focus

Continuous autofocus hunts while moving → motion blur → dropped detections at close range. A deterministic focus (drive lens position from the known tag distance) is designed but not yet deployed.

2.6  recurring  Nav2 lifecycle stall at boot

The unified bring-up launches everything at once; the navigation group comes up inactive (planner / bt_navigator / behaviour / smoother) while localization (map_server, amcl) is active → goals do nothing. Recovery: 2D Pose Estimate first, then re-trigger the lifecycle manager. Proper fix (serialise nav activation after map→odom) pending.

3. The root cause, quantified: CPU saturation

Load average 8.0 on a 4-core Pi 5 — 2× oversubscribed. Everything time-slices, and the AprilTag detector's input queue grows, so its detections go stale.

Process%CPUNote
web_video_server54 %diagnostic streamer — must be off during real runs
camera_node37 %IMX708 @ 1280×720 (software ISP)
apriltag_gate.py35 %Python re-publish of every full-res frame
apriltag_node20 %the detector itself
dock_trigger.py18 %the sequencer
Nav2 (controller/planner/bt/behaviour/collision/lifecycle/costmaps)~45 %runs idle during the visual approach

The vision input pipeline alone (camera + gate) is ~72 %. So "AprilTag is slow" is really "the Pi is 2× oversubscribed and the detector's queue backs up." Two of these are avoidable waste: the Python gate (35 % for a passthrough) and Nav2 running while unused during Phase 5 (it drives /cmd_vel directly then).

Why stale data breaks a servo (the control-theory bit). Latency is pure phase lag in the feedback loop. Past a fraction of the loop's response time it drives oscillation: the controller reacts to where the target was, overshoots, sees the error flip, over-corrects the other way — a limit cycle. For a visual servo, latency hurts far more than a bit of measurement noise (which a filter smooths). That is why fixing latency matters more than re-tuning gains.

4. Is the Raspberry Pi 5 underpowered for this?

Honest answer: borderline — and only because we run everything at full quality at once.

  • The Pi 5 (4× Cortex-A76) runs Nav2 comfortably, or a camera + AprilTag comfortably. It cannot do all of them concurrently at full resolution with headroom.
  • There is no GPU/NPU acceleration for AprilTag on the Pi — the quad detector (~1.5 cores at full res) runs on the CPU, competing with Nav2 and the camera. On accelerated platforms the same detection is <10 ms and off the CPU entirely.
  • Real-time control is already on a separate MCU (the Teensy) — that part is right.

So it is a resource-budget / system-design problem, not "the Pi is junk". With the software fixes below it can be made to work; for comfortable margins you offload vision or change the docking modality.

5. How others solve this

Compute platform

  • Serious / commercial AMRs: an x86 mini-PC (Intel NUC) for Nav2 + perception, or an NVIDIA Jetson (Orin) when vision is heavy. Jetson's isaac_ros_apriltag runs AprilTag on the GPU in <10 ms, leaving the CPU for Nav2. This is the single biggest difference from our setup.
  • Real-time control on a dedicated microcontroller (we do this).

Docking modality — most production AMRs do not use a camera + AprilTag

  • 2D-LiDAR + reflective markers (the dominant industrial approach): the existing safety LiDAR sees high-intensity returns from retro-reflectors and triangulates the dock. No camera, no GPU, robust to lighting, and it reuses a sensor we already have (RPLIDAR). Nav2 ships opennav_docking oriented this way.
  • IR beacon homing (Roomba-style powered dock), depth-camera + on-device detection (OAK cameras run the detector on-camera), or a mechanical V-guide so the vision only needs coarse accuracy.
  • Camera + AprilTag (our approach) is common in research / for flexible targets, but it is the most compute-hungry and the most lighting/focus-sensitive option.

6. Solutions — prioritized

A · Now — software, free, keep the Pi

  1. done Throttle the AprilTag input to ~10 fps (a max_fps param in the image gate). Kills the queue backlog → fresh detections, at full resolution (no tag-precision loss, unlike lowering the camera resolution).
  2. Never run the visualisation tools during a real dock (web_video_server alone was 54 % CPU).
  3. Pause / deactivate Nav2 during Phase 5. It drives /cmd_vel directly; the ~45 % nav stack is idle but running. Frees ~1.5 cores exactly when the detector needs them. Highest-value change left.
  4. Replace the Python gate with a C++/intra-process forwarder; deterministic camera focus; ROI tracking.

B · Medium — offload the vision

  • NVIDIA Jetson Orin Nano + isaac_ros_apriltag (GPU, <10 ms, off the CPU) — keeps the camera + AprilTag design with real margins.
  • OAK-D / Luxonis camera — runs the fiducial detector on-camera.

C · Best for production — change the modality

  • 2D-LiDAR reflective markers — reuses the RPLIDAR, no camera compute, no focus/lighting fragility, aligns with Nav2 opennav_docking. Most robust, lightest on compute.
  • Add a mechanical V-guide so vision only needs coarse accuracy.

Recommended path. Ship A1–A3 (throttle + kill viz + pause Nav2 during the approach) — this alone should bring the effective detection latency under ~120 ms on the current Pi and make the visual servo behave, with no new hardware. If docking must be bullet-proof for production, move to 2D-LiDAR reflectors, or offload AprilTag to a Jetson if the camera/tag design must stay.

7. One-line takeaway

The docking maths and the motor are fine. The blocker is feeding a real-time visual servo from a CPU-starved detector. Fix the resource budget — throttle vision, and stop running Nav2 and the detector flat-out at the same time — before touching the control gains again; or move the fiducial detection off the CPU / onto the LiDAR.
