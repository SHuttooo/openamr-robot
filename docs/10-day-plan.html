What I accomplished today (Day 1, 29 Jun)

The long-standing “left wheel oscillation” — a slow wobble no PID gain could remove — was diagnosed and fixed for good. By measuring the encoder, I found it was not a control problem but a sensor defect: a decentered encoder magnet reporting a ±40 % fake velocity ripple (twice per wheel turn). Fixing it properly meant rebuilding the whole low-level velocity loop. Done, tuned, flashed, committed and documented.

  • Encoder-ripple correction — a correction table loaded at runtime and re-aligned automatically in ~8 s per power-on (±40 % → ±4 %).
  • Velocity feedforward — the same, consistent response at every speed (a plain PID overshot more and more as speed rose).
  • Back-calculation anti-windup — removes the high-speed overshoot.
  • Low-speed velocity estimator — clean speed reading where the sensor is otherwise too coarse.
  • Anti-stiction dither — smooth motion down to ~0.06 m/s (needed for docking).
  • Final gains tuned & baked into firmware; full write-up + technical-report case study + this plan & Foxglove proposal.

Result: a reliable, well-documented velocity loop — smooth from ~0.06 m/s up to the navigation max, with the same behaviour at any speed.

Foxglove integration — proposal

What forA single web-based observability layer for the real robot — live monitoring and recorded-log (MCAP) playback of navigation, perception, motor control and docking. Valuable because the robot is a headless Raspberry Pi over WiFi.
Where in the workflowReal-robot bring-up, tuning & validation (Days 2–8) as the operator/debug dashboard, and for post-mortem analysis of recorded missions.
Topics / dataTF, /odom, /scan+/scan_filtered, /map, global/local costmaps + footprint, the cmd_vel chain, /camera/...compressed, Nav2 BT status, /debug/* motor telemetry, docking state + AprilTag detections. Logs: MCAP rosbags.
Why current tools fall shortRViz over WiFi needs X-forwarding/same subnet and lags with camera; Gazebo is simulation-only; rqt is fragmented and needs X; rosbag has no integrated GUI. None gives one remote web UI = 3D + plots + image + log playback.
Benefit vs RViz / Gazebo / rqt / rosbagBrowser-based (no X), efficient remote streaming, one layout (3D + time-series + camera + state), savable/shareable layouts, native MCAP playback with the same panels. Much lower friction for field debugging on a headless robot.
Repo changes requiredMinimal & additive: install foxglove_bridge, a small foxglove.launch.py (or a bringup include), optionally a saved layout JSON. No code changes to existing packages.
Same repo or separateInside the existing repo (a launch file + layout JSON under a foxglove/ folder). A thin add-on — no separate repository needed.
Deliverables & success(a) bridge launchable with the stack; (b) a saved layout (nav + sensors + motor telemetry + docking); (c) monitor a live nav+dock mission over WiFi; (d) replay a recorded MCAP. Success = debug a full mission remotely (no RViz/X) and analyze a recorded run.

  • stretch UI polish · Foxglove monitoring.
