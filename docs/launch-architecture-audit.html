1. One-page verdict

▲ Overall consistent on the REAL side (1 command = everything), but asymmetric on the SIM side: there are TWO parallel simulation worlds that each bring up Gazebo + Nav2 through different paths.
CriterionRatingDetail
Sim/real coexistence medium The selector bringup.launch.py sim:=true|false is clean and shares the same Nav2 + the same nav2_params.yaml. BUT the SIM docking is not plugged into it: it lives in a second complete launcher (bringup_sim.launch.py) that brings up Gazebo+Nav2 via another wrapper. On the REAL side, docking IS integrated (use_docking). → asymmetry.
Duplicates 2 real (a) Two competing SIM Nav2 stacks ; (b) two different map injection mechanisms. Plus 2 justified "false duplicates" (camera_info bridge, apriltag sim/real).
Clarity medium 4 ways to launch the sim coexist (selector, 1-command, 3-terminals, isolated layers). None broken, but nothing indicates which one is "the right one" → confusion (this is exactly the question that triggered this audit).
Goal routing (/goal_pose) fragile navigation_launch remaps the bt_navigator goal onto /goal_pose_nav. As a result a forwarder is REQUIRED. Two forwarders exist (relay vs dock_trigger) and must NEVER run together.

3. The 4 entry points (actual expansion trees)

A · bringup.launch.py — the selector (the integration)

The only launcher that switches sim↔real on the same Nav2. OpaqueFunction → chooses the source based on sim.

sim:=true SIM
└ gz_simulator.launch.py # Gazebo + bridge + RSP
└ Node relay /goal_pose → /goal_pose_nav FORWARDER #1
└ localization_launch.py # map='', params=temp, sim_time=true
└ navigation_launch.py # sim_time=true, use_scan_filter=true
└ rviz2 # if use_rviz
sim:=false REAL (default)
└ real_bringup.launch.py
└ drivers.launch.py · scan_body_filter · camera.launch.py · EKF · static TFs
└ if use_docking=true → docking_real.launch.py FORWARDER #2 (dock_trigger)
└ apriltag.launch.yml · detected_dock_pose · dock_trigger # dock_trigger owns /goal_pose
└ otherwise → relay /goal_pose → /goal_pose_nav FORWARDER #1
└ localization_launch.py # map='', params=temp (map_server bug fix)
└ navigation_launch.py # sim_time=false, use_scan_filter=false
⇒ On the REAL side: docking integrated via use_docking. On the SIM side: no docking, just the relay. ← the asymmetry.

B · bringup_sim.launch.py (docking) — the historical "1 entire command"

bringup_sim.launch.py full SIM + docking
└ t=0 : gz_simulator.launch.py
└ t=8s : sim_bringup_launch.py 2nd Nav2 stack
└ localization_launch.py # map=my_map.yaml passed directly
└ navigation_launch.py # use_scan_filter = default (not passed)
└ rviz2

└ parameter_bridge /camera_info 2nd bridge
└ camera_info_sync · apriltag_sim · detected_dock_pose
└ dock_trigger # owns /goal_pose FORWARDER #2
⇒ Brings up Gazebo + Nav2 as a conceptual duplicate of A(sim), but via sim_bringup_launch and with docking. Not modified by the integration.

C · 3 manual terminals (sim)

T1 └ gz_simulator.launch.py
T2 └ sim_bringup_launch.py # = same 2 includes as B, step 2
T3 └ openamrobot_docking.launch.py # = same layer as B, step 3
⇒ Exactly the contents of B "unfolded" by hand. B = C + TimerActions. Not a 4th implementation, just the same graph launched differently.

D · Isolated building blocks (debug)

Each include above can be launched on its own: real_bringup, drivers, camera, online_async_launch (SLAM), etc. Healthy — these are reusable leaves, not duplicates.

4. Duplicates & overlaps — the precise list

#ElementSeverityDescriptionRecommendation
1 Two SIM Nav2 stacks
A(sim) vs sim_bringup_launch		 medium bringup.launch.py sim:=true includes localization+navigation directly. sim_bringup_launch.py (used by B and C) includes the same two, but freezing use_sim_time=true and the "old-style" map passing. Two ways to bring up the same Nav2 in sim. Make sim_bringup_launch either a simple alias of A(sim), or deprecate it. Eventually: A must be the single source.
2 Two map injections medium A writes a temporary nav2_params.yaml with map_server.yaml_filename pre-filled (fixes the empty map_server bug). sim_bringup_launch passes map:=my_map.yaml directly (RewrittenYaml path). Two mechanisms → only one is "the fix". Unify on A's temp-params method (deterministic), or verify that the direct method suffices in sim and document why there are two.
3 Two /goal_pose→/goal_pose_nav forwarders
relay vs dock_trigger		 high* The relay (A sim, and A real if use_docking=false) AND dock_trigger (docking) both publish on /goal_pose_nav. *Not a bug as long as only one is launched. But if you launch A sim:=true (relay) + the sim docking layer → double publication of the goal. Exclusivity by construction: when docking runs, remove the relay. This is already the case on real (use_docking); to be reproduced in sim.
4 parameter_bridge ×2
gz_simulator + openamrobot_docking		 false duplicate The main bridge is in gz_simulator. openamrobot_docking launches a 2nd one dedicated to /camera_info (root) because the main bridge only publishes /camera/camera_info, not where image_transport looks. Justified by a comment in the code. OK as is. Ideally move the camera_info bridge into gz_simulator to centralize everything.
5 apriltag sim vs real
apriltag_sim.launch.yml / apriltag.launch.yml		 healthy Two files because the camera topics differ (/rgb_image sim vs /camera/image_raw real). A genuine sim/real split, not a duplicate. Nothing to do. Good separation.
6 use_scan_filter not passed by sim_bringup minor A(sim) forces use_scan_filter=true. sim_bringup_launch does not pass the argument → it depends on the navigation_launch default. Potentially different filtering behavior between B and A in sim. Align: explicitly pass the same value in sim_bringup.

The Nav2 core coexists perfectly (a single code, a single params file). The only real fracture is docking: integrated into the selector on real, but isolated in a separate launcher on sim.

6. What should be done (if we want a symmetric "1 command")

  1. Plug the sim docking into the selector: add use_docking to the sim:=true branch of bringup.launch.py (include openamrobot_docking.launch.py instead of the relay, like on real with docking_real). → removes duplicate #3 in sim and the asymmetry.
  2. Reduce bringup_sim.launch.py to a thin alias that calls bringup.launch.py sim:=true use_docking:=true (keeps the historical command for compatibility, but a single implementation behind it). → removes duplicate #1.
  3. Unify the map injection (#2): sim_bringup and A must use the same mechanism.
  4. Align use_scan_filter (#6).

⚠️ The user explicitly said they do not want to deviate too much from the old version. So: keep B and C functional (don't break anything), and add symmetry only additively (new argument, alias) — no removal.
