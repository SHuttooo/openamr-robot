## Foxglove integration — proposal (for review before any structural change)

> Requested: justify Foxglove before integrating. **No repo changes will be made until this is approved.**

- **What we want to use it for.** A single **web-based observability layer** for the *real* robot — live
  monitoring **and** recorded-log (MCAP) playback of navigation, perception, motor control and docking.
  Especially valuable because the robot is a **headless Raspberry Pi over WiFi**.

- **Where in the OpenAMR workflow.** During real-robot Nav2/docking bring-up, tuning and validation
  (Days 2–8) as the operator/debug dashboard, and for **post-mortem analysis** of recorded missions.

- **Topics / data to inspect.** TF tree; `/odom`; `/scan` + `/scan_filtered`; `/map`;
  `/global_costmap` + `/local_costmap` (+ footprint); the cmd_vel chain
  (`/cmd_vel_nav → /cmd_vel_smoothed → /cmd_vel`); `/camera/image_raw/compressed`; Nav2 BT status /
  feedback; **`/debug/left|right|pwm`** (motor telemetry — directly useful for the PID/feedforward work);
  docking state + AprilTag detections. **Logs:** MCAP rosbags for replay.

- **Why current tools are not sufficient.** RViz over WiFi needs X-forwarding or a same-subnet desktop
  and lags badly with camera; **Gazebo is simulation-only** (no real-robot data); **rqt** is fragmented
  (one tool per concern) and also needs X; **rosbag/ros2 bag** has no integrated GUI. None offers a
  single remote web UI combining 3D + time-series + image + log playback.

- **Benefit vs RViz / Gazebo / rqt / rosbag.** Browser-based (no X), efficient remote streaming, **one
  layout** = 3D scene + plots (ideal for odom/velocity/PID debugging) + camera + state; **savable,
  shareable layouts**; native **MCAP playback** with the same panels used live. Much lower friction for
  field debugging on a headless robot.

- **Required changes (minimal, additive).** Install `foxglove_bridge`
  (`ros-jazzy-foxglove-bridge`); add a small `foxglove.launch.py` (or an include in the bringup);
  optionally store a saved **layout JSON** and an MCAP-recording config in the repo. **No code changes to
  existing packages.**

- **Same repo or separate.** **Inside the existing repo** — a launch file + layout JSON under a
  `foxglove/` folder (or the bringup package). It is a thin observability add-on; **no separate
  repository needed.**

- **Deliverables & success criteria.** (a) `foxglove_bridge` launchable alongside the stack; (b) a saved
  layout showing nav + sensors + motor telemetry + docking; (c) monitor a **live** real nav+dock mission
  over WiFi; (d) **replay** a recorded MCAP with the same panels. *Success = an operator can observe and
  debug a full mission remotely (no RViz/X) and analyze a recorded run afterward.*

- **Process.** Proposal only, today. Integration scheduled **Day 7**, contingent on approval.

### §A — Reference: what was completed Day 1 (29/06)
Motor velocity loop fully rebuilt and tuned (the "left wheel oscillation" was a decentered encoder
magnet → ±40 % 2/rev measurement ripple, not a control problem): runtime encoder-ripple correction
table (re-aligned ~8 s/boot), velocity feedforward (`PWM = Kff·target + PID`, same response at every
speed), back-calculation anti-windup, low-speed velocity estimator, anti-stiction dither (smooth to
~0.06 m/s). Gains baked: Kp 2.0 / Ki 0.10 / Kd 0.10, Kff 7.87, dither 92. Full write-up:
`docs/history/encoder-calibration.md`; report case study `docs/history/rapport-stage-technique.md` §5.1.
